# Mobile UX polish and admin-editable design images

This change addresses two related themes: making the site comfortable to use on mobile devices (touch targets, full-width buttons, swipe gestures, scroll overflow), and wiring EditableImage into the design catalog so the admin can replace photos on static designs without touching code. The About section default image is swapped (pexels 3852974 -> 3738088), and Logo letter-spacing is tightened.

**Watch for:** Swipe-down-to-close on the Lightbox conflicts with the scrollable container on mobile (confirmed). EditableImage drops `loading="lazy"` for all static catalog images (confirmed). Custom designs lose error fallback in the Lightbox (confirmed).

## High-level view

The Lightbox gained a `drag="y"` gesture (Framer Motion) so users can swipe down to dismiss, plus a mobile hint bar, a bottom "Close" button, and `max-h-[90vh] overflow-y-auto` to make content scrollable. These two features — draggable container and scrollable container — fight each other on mobile because Framer Motion's drag handler intercepts vertical touch events that the user might intend as scrolls.

DesignCard now branches on `design.custom`: custom (admin-uploaded) designs keep the original `<img>` with `loading="lazy"` and an `onError` fallback, while static catalog designs render `EditableImage`. This means static designs lose lazy-loading entirely — EditableImage renders a plain `<img>` without `loading="lazy"`. The Lightbox uses EditableImage with the same `contentKey` pattern (`design.image.${design.id}`), so an admin photo replacement propagates to both views correctly — but the Lightbox has no error fallback for broken image URLs.

The global `overflow-x: hidden` on `<html>` in index.css prevents any accidental horizontal scroll but hides symptoms of overflowing elements rather than fixing the root cause. The Contact.tsx `break-all` fix in this same change is the correct approach for the known overflow.

<details>
<summary>Issues (4)</summary>

1. **Swipe-vs-scroll conflict in Lightbox** — `drag="y"` on the same element that has `overflow-y-auto` creates ambiguous touch behavior on mobile. Framer Motion will intercept vertical swipes even when the user wants to scroll the lightbox content. Consider restricting drag to only fire when the scroll position is at the top, or move the drag handle to the hint bar only.
2. **Lazy-loading dropped for static designs** — EditableImage renders `<img>` without `loading="lazy"`, so all static catalog images will eager-load. On a page with 20+ designs, this hurts initial page load on mobile connections. Add a `loading` prop to EditableImage.
3. **overflow-x: hidden on html masks layout bugs** — This hides symptoms rather than fixing the element(s) that overflow. If a future component overflows, the content will be silently clipped with no visual signal.
4. **Custom designs lose error fallback in Lightbox** — The Lightbox uses EditableImage for all designs (including custom), but EditableImage has no `onError` fallback. A 404'd custom image shows a broken image in the lightbox while DesignCard gracefully shows a Shirt icon.

## Swipe-to-close vs scrollable content in Lightbox

The motion.div that wraps lightbox content now has both `drag="y"` (with `dragConstraints={{ top: 0, bottom: 0 }}` and `dragElastic={0.4}`) and `max-h-[90vh] overflow-y-auto`. On mobile, any vertical touch movement is ambiguous: is the user scrolling inside the lightbox or swiping to dismiss?

Framer Motion's drag handler captures the touch start. With `dragElastic={0.4}`, the container moves visually before the user's intent is clear. The `onDragEnd` threshold of 100px offset is generous enough that a fast scroll might not trigger close, but the visual jank of the container shifting during a scroll attempt degrades the experience.

```tsx
drag="y"
dragConstraints={{ top: 0, bottom: 0 }}
dragElastic={0.4}
onDragEnd={(_e, info) => {
  if (info.offset.y > 100) onClose();
}}
// ... combined with:
className="... max-h-[90vh] overflow-y-auto ..."
```

A common fix: only enable `drag` when `scrollTop === 0`, or attach the drag gesture to the hint bar element alone rather than the entire modal.

**Confidence: confirmed** — both `drag="y"` and `overflow-y-auto` are on the same element (Lightbox.tsx lines 78 and 85).

## EditableImage loses lazy-loading for the static catalog

`DesignCard` previously rendered all images with `loading="lazy"`. The refactor splits behavior: custom designs still get the original `<img loading="lazy" onError={...}>`, but static designs now render `EditableImage` which internally renders `<img src={currentSrc} alt={alt} className={className} />` — no `loading="lazy"` attribute.

On a gallery page showing 20+ designs, every static design image will start loading immediately rather than deferring until near-viewport. On slow mobile connections this is a meaningful regression for a change focused on mobile UX.

The fix: EditableImage should accept and pass through a `loading` prop.

**Confidence: confirmed** — EditableImage.tsx line 76 renders `<img>` without `loading`.

## Lightbox uses EditableImage unconditionally

The Lightbox renders EditableImage for all designs regardless of `design.custom`:

```tsx
<EditableImage
  contentKey={`design.image.${design.id}`}
  defaultSrc={design.image}
  alt={design.name}
  className="h-full w-full object-cover"
/>
```

For custom designs, the image URL may point to a user-uploaded file that could 404. EditableImage renders a bare `<img>` with no error handling — the user sees a broken image. DesignCard handles this gracefully with `onError` + a Shirt icon fallback, but that path is bypassed when the lightbox opens.

**Confidence: confirmed** — Lightbox.tsx lines 112-116 vs DesignCard.tsx lines 69-76.

## Global overflow-x: hidden on html

```css
html {
  scroll-behavior: smooth;
  overflow-x: hidden;
}
```

The body already had `overflow-x: hidden`. Adding it to `<html>` as well ensures nothing can escape, but it's a debugging blanket. The Contact.tsx `break-all` fix in this same PR addresses the known overflow issue properly. The html-level override makes future overflow bugs harder to notice since there's no scrollbar signal.

**Confidence: likely** — no specific overflow bug was identified that requires the html-level fix beyond what Contact.tsx already solves.
