# Security hardening for Netlify-hosted React SPA

A comprehensive security pass over the Happiness Fashion World portfolio site. Adds HTTP security headers via netlify.toml (CSP, HSTS, X-Frame-Options, COOP, CORP, Permissions-Policy), a CSP meta tag fallback in index.html, a new `sanitize.ts` utility module (text stripping + image magic-byte validation), rate limiting on the admin passcode (5 attempts, 5-minute lockout), image upload validation with file-type enforcement, and input sanitization wired into EditableText, EditableImage, and SiteContentContext.

**Watch for:** The `sanitizeText` function decodes HTML entities *after* stripping tags, which re-introduces characters that could form injection payloads when the output is later inserted via `dangerouslySetInnerHTML` or rendered without React's escaping (confirmed). Rate limiting state lives in `sessionStorage` — closing and reopening the browser tab resets the lockout entirely, making it trivial to bypass (confirmed). The CSP blocks the Vite development server's WebSocket HMR connection (`ws://localhost:*`), so `npm run dev` will break with console errors unless developers manually relax the policy or Vite injects its own meta tag override (likely).

## High-level view

The CSP policy is well-scoped for production: script-src restricted to 'self', img-src covers Pexels CDN and Supabase Storage, connect-src allows Supabase and Gemini API. The policy is duplicated identically in netlify.toml and index.html (meta tag fallback). The `style-src 'unsafe-inline'` allowance is necessary because Framer Motion injects inline styles at runtime — this cannot be tightened without a major refactor. However, the meta tag's `connect-src 'self'` blocks Vite's dev-server WebSocket, breaking HMR during local development.

The sanitization layer applies defense-in-depth: `sanitizeText` runs both at the component level (EditableText.save) and at the storage layer (SiteContentContext.set). The entity-decode step converts `&lt;` back to `<` *after* stripping tags — if that output were ever passed through HTML rendering without escaping, the decoded characters form new tags. React's JSX escapes by default so this is safe today, but the function's contract is a latent footgun for any future non-React consumer.

The rate limiter is a UX deterrent, not a security boundary. `sessionStorage` is per-tab and clears on tab close. The underlying passcode `'happy2026'` remains a hardcoded string literal in the client bundle. For this app's threat model (a single-owner fashion portfolio), the rate limiter adds friction for casual guessing, which is adequate.

Image validation checks magic bytes for JPEG, PNG, WebP, and HEIC/HEIF before the file reaches the resize pipeline. One inconsistency: `EditableImage` logs validation failures to console without showing user-facing feedback, while `AddDesignPanel` correctly displays the error message.

## CSP policy and the dev-server gap

The CSP meta tag in `index.html` is served during both production and development. Vite's dev server creates a WebSocket connection for HMR at `ws://localhost:5173/__vite_hmr`. The `connect-src 'self'` directive blocks this, producing console CSP errors and broken hot reload.

Two fix options:
- Remove the CSP meta tag from `index.html` and rely solely on the netlify.toml header (only applies in production on Netlify).
- Use a Vite `transformIndexHtml` plugin to strip the meta tag in development mode.

The second option preserves the meta tag fallback for non-Netlify deployments while not breaking local dev.

## Sanitization: entity decode ordering concern

```ts
// sanitize.ts lines 22-35
let clean = input
  .replace(/<script\b[^<]*(?:(?!<\/script>)<[^<]*)*<\/script>/gi, '')
  .replace(/<style\b[^<]*(?:(?!<\/style>)<[^<]*)*<\/style>/gi, '')
  .replace(/<[^>]*>/g, '')  // strip all tags
  // Then decode entities:
  .replace(/&amp;/g, '&')
  .replace(/&lt;/g, '<')
  .replace(/&gt;/g, '>')
```

Consider input: `&lt;script&gt;alert(1)&lt;/script&gt;`. After tag stripping (no actual tags present), it remains unchanged. Then entity decode converts it to `<script>alert(1)</script>`. The function returns a string containing literal `<script>` tags.

This is safe *only* because React renders through JSX interpolation which HTML-escapes. But `sanitizeText` is exported as a general utility — any future consumer that uses the output in `dangerouslySetInnerHTML`, a template literal, or a server-rendered context would be vulnerable. The safer approach: decode entities *first*, then strip tags from the decoded result. Or remove the decode step entirely, since the only consumer is React which handles escaping itself.

## Rate limiting: sessionStorage scope

The lockout state persists in `sessionStorage` under key `'hfw-admin-attempts'`:

1. `sessionStorage` is per-tab: opening a fresh tab to `/#admin` starts with zero attempts and no lockout.
2. `sessionStorage` clears when the tab is closed: closing and reopening achieves the same reset.

For a client-side-only passcode check (no server round-trip), there is no storage mechanism that fully prevents counter resets. `localStorage` would survive tab close but is still clearable via DevTools. The implementation is appropriate as a UX friction layer for this threat model — the real security boundary is that the passcode only unlocks local admin UI with no server-side privileges.

## EditableImage silent validation failure

When `validateImageFile` returns `{ valid: false }` in `EditableImage.onFile`:

```ts
const validation = await validateImageFile(file);
if (!validation.valid) {
  console.warn('[EditableImage] invalid file:', validation.error);
  return;  // No user-facing feedback
}
```

The admin sees the camera overlay return to its default state with no indication of what went wrong. Adding a toast notification or inline error state (similar to `AddDesignPanel`'s `setError`) would make this consistent.
