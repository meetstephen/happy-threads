All five concerns from the v1 review have been addressed. Entity decoding now runs before tag stripping in `sanitizeText`, the CSP meta tag was removed from `index.html` (relying solely on Netlify's response header), double sanitization was eliminated by removing `sanitizeText` from `SiteContentContext`, and `EditableImage` now surfaces validation errors to the admin. The rate limiter's `sessionStorage` scope was acknowledged as appropriate for the client-side threat model.

## High-level view

The sanitization fix correctly reorders operations: HTML entities are decoded first, then any resulting tags (including reconstituted `<script>` from `&lt;script&gt;`) are stripped. The function's output is now safe regardless of whether the consumer HTML-escapes it or not â€” a genuine improvement over the v1 approach.

The CSP strategy is now clean: a single policy defined in `netlify.toml` that only takes effect in production. Local development with Vite is unaffected since there is no meta tag to conflict with the dev server's WebSocket. The policy itself is well-scoped â€” `script-src 'self'` without `unsafe-inline` or `unsafe-eval`, `object-src 'none'`, `frame-src 'none'`, and explicit allowlists for the third-party origins the app actually uses (Pexels CDN, Supabase, Gemini API).

Sanitization is now single-pass: `EditableText` calls `sanitizeText` at save time, and `SiteContentContext` stores the result as-is. This eliminates the truncation edge case where two passes could shorten content unexpectedly.

The rate limiter's `sessionStorage` scope is an inherent limitation of client-side enforcement. For a fashion portfolio where the passcode unlocks a local admin UI (no server-side privileges), this is adequate friction.

## Entity decode ordering fix

The v1 review identified that decoding entities after stripping tags would reconstitute `<script>` from `&lt;script&gt;`. The current implementation reverses the order:

```ts
let clean = input
  // Decode common HTML entities FIRST
  .replace(/&amp;/g, '&')
  .replace(/&lt;/g, '<')
  .replace(/&gt;/g, '>')
  .replace(/&quot;/g, '"')
  .replace(/&#x27;/g, "'")
  .replace(/&#x2F;/g, '/')
  // THEN remove script/style/all tags
  .replace(/<script\b[^<]*(?:(?!<\/script>)<[^<]*)*<\/script>/gi, '')
  .replace(/<style\b[^<]*(?:(?!<\/style>)<[^<]*)*<\/style>/gi, '')
  .replace(/<[^>]*>/g, '')
```

Input `&lt;script&gt;alert(1)&lt;/script&gt;` now decodes to `<script>alert(1)</script>`, which the tag-stripping regex then removes entirely â€” producing `alert(1)` as plain text. The function is safe even if its output is used in a non-React context.

## EditableImage validation feedback placement

The error is rendered inside the overlay that uses `opacity-0 group-hover/img:opacity-100`:

```tsx
<span className="absolute inset-0 flex items-center justify-center gap-2 bg-ink-900/50 opacity-0 transition-opacity group-hover/img:opacity-100">
  {/* ... buttons ... */}
  {validationError && (
    <span className="mt-2 block rounded-lg bg-wine-500/10 px-3 py-2 text-[11px] text-wine-500">
      {validationError}
    </span>
  )}
</span>
```

The admin triggers the file picker (which requires hovering to see the button), selects an invalid file, and the error appears in the same overlay. Since the file picker dialog forces the cursor away from the image, the overlay may briefly hide. On mouse-return, the error reappears and persists until the next upload attempt clears it. This is functional but slightly awkward â€” not a blocking concern.
