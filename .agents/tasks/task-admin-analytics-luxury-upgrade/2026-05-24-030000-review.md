# Admin analytics, luxury polish, and expanded catalog

A large multi-feature upgrade to the Happiness Fashion World portfolio site. Adds a Supabase-backed analytics dashboard (page views, section engagement time, design views/likes) gated behind the admin panel. Expands the design catalog from 12 to 20 entries with Nigerian fashion focus. Rewrites the Joy AI chatbot for warmer, more human language with 8+ new intents. Introduces a "What We Sew" section, a floating sewing icons background layer, and luxury CSS utilities (border-shimmer, gold-glow, pulse-gold). Mobile-optimizes the admin panel with full-viewport layout and touch-friendly controls.

**Watch for:** Analytics event flush on `beforeunload` calls an async function in a synchronous event handler — events can be silently dropped on page close (confirmed). The analytics tab is accessible with the local hardcoded passcode, not just Supabase-authenticated admin — anyone who knows "happy2026" can view analytics data stored locally (confirmed). `initAnalytics` registers a new `beforeunload` listener every time it's called with no idempotency guard; in strict-mode dev (double-mount), this registers two listeners (likely).

## High-level view

The analytics system is a lightweight event queue that batches writes to Supabase (or localStorage as fallback), paired with an admin dashboard that aggregates and renders bar charts and rankings. IntersectionObserver tracks section dwell time, favorites context changes track likes, lightbox opens track views. All section IDs in the analytics observer match the actual DOM IDs. The provider tree in main.tsx correctly wraps FavoritesProvider around App so the `useFavorites` hook resolves. The main gap is the `beforeunload` flush path and the lack of human-readable design names in the dashboard.

The expanded catalog (HF-013 through HF-020) follows the existing `Design` interface — categories are valid union members, `pexels()` helper used consistently.

The chatbot rewrite preserves the `BotMessage` interface and `generateReply` export signature. New intent patterns are prepended to the rules array, correctly matching before the broad "show a design" catch-all.

The Supabase RLS policies for the analytics table have the correct asymmetry: anonymous INSERT, admin-only SELECT. However, this protection only applies to the Supabase path — the localStorage fallback exposes all analytics data to anyone with the local passcode. If Supabase is configured but the user unlocks via the local passcode (no JWT), the RLS SELECT policy will deny the query, and the dashboard silently shows "Could not load analytics data."

1. **Async flush in beforeunload** — `flush()` is async (awaits Supabase insert) but `beforeunload` is synchronous; the browser may kill the page before the promise resolves. Use `navigator.sendBeacon` or synchronous localStorage write for the final flush.
2. **initAnalytics is not idempotent** — Each call adds a new `beforeunload` listener. React strict-mode double-mounts will register two. Add a guard (`let initialized = false`) or use `{ once: true }` / `removeEventListener`.
3. **Analytics tab accessible via local passcode** — The hardcoded `LOCAL_PASSCODE = 'happy2026'` is in the client bundle and grants access to the Analytics tab. Anyone who reads the source or guesses the code sees analytics data (from localStorage). The acceptance criteria say "admin-only" but the local fallback has no real auth.
4. **Analytics dashboard shows design IDs, not names** — `AnalyticsDashboard` renders `design.id` (e.g., "HF-003") in the Top Designs list. The `name` field in the aggregated data is also just the ID. Resolving to human-readable names requires passing the designs array to the dashboard or joining at query time.

## Analytics event lifecycle and the beforeunload gap

The analytics service queues events in memory and flushes every 5 seconds via `setTimeout`. On page unload, `initAnalytics` registers a `beforeunload` listener that calls `flush()`. The problem: `flush` is `async` — it awaits `supabase.from('site_analytics').insert(rows)`. The `beforeunload` handler is synchronous; the browser tears down the page before the await resolves.

```ts
// analytics.ts line 78-83
export function initAnalytics() {
  window.addEventListener('beforeunload', () => {
    flush(); // async, return value ignored — promise may never settle
  });
}
```

For the Supabase path, `navigator.sendBeacon` with a JSON payload (or `fetch` with `keepalive: true`) would survive page teardown. For the localStorage fallback, a synchronous `localStorage.setItem` inside `beforeunload` works fine. The current code already uses localStorage in the catch branch, but the primary Supabase path is the one that drops events.

Additionally, `initAnalytics` has no idempotency guard. React 18 strict mode double-invokes the effect, registering two `beforeunload` listeners. In production (single mount) this is benign, but it's a latent bug that surfaces as duplicate flush attempts in development.

## Analytics dashboard data resolution

The `getAnalytics` aggregation produces `topDesigns` where each entry's `name` is set to `design.id` (analytics.ts ~line 166: `name: id`). The AnalyticsDashboard component renders `{design.id}` directly in the ranking list. The admin sees "HF-003" instead of "Igbo Bridal Blouse & Wrapper". The design catalog is available in the component's render context (it could import `designs` from `../data/designs` and resolve names), but this wiring isn't done.

## Security posture: local passcode exposes analytics tab

The admin panel has two auth paths: Supabase email auth (when configured) and a hardcoded local passcode (`happy2026`). The local passcode is a string literal in the shipped JavaScript bundle — trivially discoverable via DevTools or source maps. Once unlocked via either path, the Analytics tab is visible and functional.

The practical risk is low for localStorage data (it's per-browser anyway), but it creates a confusing failure mode: if Supabase is configured and the admin uses the local passcode instead of email sign-in, the dashboard's Supabase query fails silently (no JWT → RLS denies SELECT) and shows "Could not load analytics data" with no explanation of why.
