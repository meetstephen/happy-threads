# Security hardening, UX polish, and mobile bottom navigation

This change addresses three concerns in a single branch: moving the Gemini API key from the client bundle to a server-side Netlify Edge Function proxy, removing hardcoded admin credentials, and overhauling the mobile navigation experience with a fixed bottom nav bar, scroll-to-top button, loading skeletons, session-persisted chat, and text-overflow fixes. The key exposure was the highest-risk item and the proxy pattern is the correct fix for a Netlify-deployed SPA.

**Watch for:** The edge function proxy forwards arbitrary payloads to Gemini without validation or rate limiting — any internet user can POST to `/api/gemini` and consume the project's API quota (confirmed). The `isGeminiEnabled` constant is now hardcoded `true`, so every first message attempt hits the network even when the key is genuinely not configured, adding latency before fallback (confirmed). Session storage persistence of chat history has no size guard and could silently fail on quota (likely).

## High-level view

The Gemini API key is no longer bundled into the client. A Netlify Edge Function at `/api/gemini` reads the key from `Deno.env` and proxies requests to Google's API. The CSP `connect-src` was tightened to remove the direct Google domain — the browser only talks to same-origin now. However, the proxy is publicly reachable without authentication or rate limiting, so anyone who discovers the endpoint can burn through the project's Gemini quota.

Admin access was decoupled from a hardcoded passcode. `AddDesignPanel` now reads from `VITE_ADMIN_PASSCODE` and shows a helpful message when the variable is unset. The personal email was moved out of `constants.ts` into an env var with a generic fallback, while `SUPABASE_SETUP.md` documents the real email for database policies.

The mobile navigation layer adds a fixed 5-item bottom bar (hidden on `md:` and above), a scroll-to-top button, and repositions the WhatsApp FAB and chatbot launcher above the new bar. The chatbot now persists its Gemini history and messages to `sessionStorage`, surviving page reloads within a tab. The StyleQuiz was fixed to score custom (cloud) designs alongside static ones.

## Gemini proxy: open relay risk

The edge function at `netlify/edge-functions/gemini-proxy.ts` blindly forwards any POST body to the Gemini API with the server-side key appended. There is no `Origin` header check, no referrer validation, no authentication, and no rate limiting. Since Netlify Edge Functions respond to any HTTP client (not just the site's own JS), any external script can POST crafted payloads to `https://<site>/api/gemini` and consume the project's free-tier quota (15 RPM, 1M tokens/day). On the free tier the impact is limited to a denial of service for legitimate users, but on a paid plan this would have cost implications.

A minimal fix is to check the `Origin` header against the site's hostname and reject other origins. A defense-in-depth approach would also parse the JSON body and reject requests that don't match the expected shape (must contain `system_instruction`, `contents` array, `generationConfig`).

```typescript
// Example origin guard
const origin = request.headers.get('origin');
if (!origin || !origin.endsWith('.netlify.app')) {
  return new Response(JSON.stringify({ error: 'Forbidden' }), { status: 403, ... });
}
```

## Always-on Gemini flag and fallback latency

Setting `isGeminiEnabled = true` unconditionally means the chatbot always attempts a network call to `/api/gemini` before falling back to the local pattern matcher. When the key isn't configured server-side, the first message incurs a full round-trip to the edge function (which returns 503) before the catch block triggers the fallback. The UX could be tighter if the component cached the 503 result and skipped subsequent network attempts for the session — a simple `useRef<boolean>` flag set on the first 503 would eliminate repeated wasted round-trips.

## sessionStorage persistence unbounded growth

`geminiHistory` grows linearly with conversation length (each turn includes full message text), and `sessionStorage` is capped at ~5 MB per origin. A very active user in a long session could hit this limit. Since the catch is silent, the user would lose history mid-conversation without feedback. Consider capping the persisted history to the most recent N turns (e.g. 40), or trimming the oldest entries when serialized size exceeds a threshold.

## Mobile bottom nav z-index coordination

`MobileBottomNav` renders at `z-30` and is hidden on `md:` breakpoint. The WhatsApp FAB, chatbot launcher, and ScrollToTop are also `z-30`, positioned at `bottom-20` / `bottom-[5.5rem]` on mobile to sit above the 64px nav bar. On very narrow screens (320px) the WhatsApp button (`right-4`) and ScrollToTop (`right-4`, same position) could stack or overlap — worth a visual check on 320px viewport width (possible).
